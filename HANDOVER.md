# AI Handover Document: kintai-sync

This document provides a technical summary and critical verification points for the next AI agent or developer taking over the `kintai-sync` repository.

## 1. Project Essence

`kintai-sync` is a robust, serverless automation system that synchronizes attendance reports from Slack to Jobcan, Google Calendar, and Slack Status.

- **Trigger**: A single Slack message (e.g., in a department channel).
- **Brain**: Vertex AI (Gemini 3 Flash Preview) for parsing natural language.
- **Backbone**: Google Cloud (Cloud Run, Cloud Tasks, Firestore).

## 2. Modern Workflow & Tooling

The project adheres to high engineering standards using modern Python and DevOps tools:

- **Package Management**: Uses **`uv`**. All dependencies are in `pyproject.toml` and locked in `uv.lock`. Use `uv sync` to setup.
- **Orchestration**: Everything is wrapped in a **`Makefile`**.
  - `make setup`: Bootstraps the GCP project (GCS bucket for state, Terraform SA, IAM roles).
  - `make deploy`: Deploys the entire infrastructure via Terraform.
  - `make test`: Runs `pytest` with coverage (currently **90%**).
  - `make register-user`: Interactive CLI to register staff passwords in Secret Manager.
  - `make destroy` vs `make destroy-all`: Separation between app cleanup and total project reset.

## 3. Configuration Architecture

A "Configuration-over-Code" approach is strictly enforced:

- **Global Config**: `config.yaml` contains static settings (regions, AI model names, Slack formats). Loaded via `src/config.py`. **No hardcoded strings in source.**
- **User Master Data**: Google Spreadsheet (SSoT). Synced to **Firestore** via `src/sync.py` for performance and persistence.
- **Secrets**: Slack tokens and Jobcan passwords are stored in **Secret Manager**. Naming convention for passwords: `JOBCAN_PASSWORD_{staff_code}`.

## 4. Key Architectural Nuances

- **Idempotency**: `src/history.py` tracks task status in Firestore using `client_msg_id` to prevent double-posting or double-application.
- **Asynchronous Execution**:
  - `receiver.py`: Lightweight HTTP endpoint that verifies Slack signatures and enqueues to Cloud Tasks.
  - `main.py` (Worker): Heavy lifting (AI parsing, Playwright scraping, API calls). Triggered by Cloud Tasks.
- **Scalability**: Designed for multi-user support from day one. User-specific working hours are dynamically applied during processing.

## 5. Verification Points (Critical)

When making changes, ensure:

1. **Test Suite**: `make test` must pass 100%. Coverage should not drop below 90%.
1. **Mocking**: External APIs (Vertex AI, Google Calendar, Slack) must be mocked in unit tests to avoid real-world side effects.
1. **Config**: New features must add parameters to `config.yaml` rather than hardcoding.
1. **Makefile Alignment**: If adding a new lifecycle step, wrap it in a `make` command to keep `gcloud` complexity hidden.
