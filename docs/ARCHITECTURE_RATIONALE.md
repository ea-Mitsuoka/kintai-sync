# アーキテクチャ採用の理由 (Architecture Rationale)

本システム `kintai-sync` が、なぜ現在の構成（Cloud Run, Cloud Tasks, Firestore, Gemini API等）を採用しているのか、その技術的な背景と理由を解説します。

______________________________________________________________________

## 1. 処理の確実性と「Slack 3秒ルール」の回避

### 課題

SlackのWebhook（Event API）は、**3秒以内**にレスポンスを返さないと、タイムアウトとして処理を再送（リトライ）してしまいます。しかし、ジョブカンのスクレイピングやGemini APIの解析には3秒以上かかることが一般的です。

### 解決策：非同期分散処理（Cloud Tasks の採用）

- **仕組み**: 最初の受信機（Receiver）は「受け付けました」と3秒以内にSlackへ即答し、実際の重い処理は **Cloud Tasks** というキュー（待ち行列）に放り込みます。
- **理由**: これにより、Slack側でのタイムアウトを防ぎつつ、実際の処理はバックグラウンドで着実に実行できます。

______________________________________________________________________

## 2. 失敗に強いシステム（冪等性とチェックポイント）

### 課題

外部サービス（ジョブカンやGoogleカレンダー）への通信は、ネットワークエラーなどで時々失敗します。その際、単純に最初からやり直すと「ジョブカンの申請は終わっているのに、Slack通知でこけたから、リトライで二重に申請してしまう」という事故が起きます。

### 解決策：Firestore による状態管理

- **仕組み**: 各タスク（ジョブカン、カレンダー、Slack等）の完了状態を **Firestore** に記録します。
- **理由（なぜログ検索ではないのか）**:
  - **信頼性**: ログの検索は反映に時間がかかることがあり、リトライ時に「実は成功していた」ことを見逃すリスクがあります。
  - **一貫性**: Firestoreは書き込んだ直後に確実に読み取れる（強一貫性）ため、「どこまで成功したか」を正確に把握して、続きから再開できます。

______________________________________________________________________

## 3. 誰でも使える管理画面（運用の民主化）

### 課題

ユーザー設定（午前休の時間定義など）をコードやデータベース（Firestore）で直接管理すると、エンジニア以外が内容を修正できなくなります。

### 解決策：Google スプレッドシート ＋ Firestore 同期

- **仕組み**: マスターデータは使い慣れた **スプレッドシート** で管理し、システムは実行速度を上げるためにそのコピー（キャッシュ）を **Firestore** から読み取ります。
- **理由**:
  - **利便性**: 非エンジニアでもブラウザから簡単に設定変更が可能。
  - **速度**: 実行時に毎回スプレッドシートAPIを叩くと遅いため、Firestoreに同期しておくことで高速なレスポンスを実現しています。

______________________________________________________________________

## 4. 高度な自然言語解析の容易化

### 課題

「明日の午前休」「来週月曜の全休」など、人によって書き方が異なるSlackメッセージを、正確にプログラムで解析するのは非常に困難です。

### 解決策：Vertex AI (Gemini API) の採用

- **理由**: 従来の正規表現（文字のパターンマッチング）では限界があった複雑な文章も、LLMを使うことで、極めて高い精度で「日付・種別・理由」を構造化データとして抽出できます。

______________________________________________________________________

## 5. コストの最小化と管理の自動化

### 課題

サーバーを立てると、使っていない時間も費用が発生し、管理の手間もかかります。

### 解決策：サーバーレス ＋ Terraform

- **理由**:
  - **サーバーレス (Cloud Run / Firestore)**: 今回の規模（1日15人程度）では、ほとんどのリソースが **無料枠** に収まり、月額費用をほぼ $0 に抑えられます。
  - **Terraform (IaC)**: インフラをコードで定義することで、誰の環境でも同じ構成を再現でき、不要になったら `terraform destroy` でボタン一つで全ての課金リソースを確実に消去できます。

______________________________________________________________________

## 結論

このアーキテクチャは、\*\*「Slackの制約（3秒）を回避し、外部連携の失敗をFirestoreで確実にリカバーしつつ、非エンジニアがスプレッドシートで楽に管理できる」\*\*という、実運用での堅牢性と利便性を最優先に設計されています。
